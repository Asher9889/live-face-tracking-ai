# Event Publishing & Streams Architecture

## 📋 Summary

Your system now has **two-tier event delivery**:
1. **Pub/Sub** - Real-time subscribers only (fast, no history)
2. **Streams** - Event history + consumer groups (replay-capable)

---

## 🎯 Event Classification

### ✅ ALLOWED_EVENTS (Pub/Sub + Stream)
```python
ALLOWED_EVENTS = {
    "person_entered",      # Known person entered gate
    "person_exited",       # Known person exited gate
    "unknown_entered",     # Unknown person entered gate
    "unknown_exited",      # Unknown person exited gate
}
```

**Delivery:**
```
EventPublisher.publish(event_type, payload)
    ├─→ Redis Pub/Sub: live-face-tracker:camera-events:{camera_code}
    └─→ Redis Stream: live-face-tracker:stream:camera-events:{camera_code}
```

### ❌ OTHER EVENTS (Silently Filtered)
```python
OTHER_EVENTS = {
    "face_detected",           # Face quality passed threshold
    "recognition_pending",     # Waiting for face recognition
    "recognition_confirmed",   # Identity confirmed (internal only)
    "unknown_confirmed",       # Unknown created (internal only)
    "track_created",          # New person detected (commented out)
    "track_updated",          # Person bbox updated (commented out)
}
```

**Current:** These are generated by `TrackEventEmitter` but silently dropped by `EventPublisher.publish()`.

**If you need them:** Add to `ALLOWED_EVENTS` or create separate stream keys.

## 🏗️ Event Flow Architecture

```
Camera Detection
    ↓
YOLO Tracker
    ↓
TrackEventEmitter (app/tracking/track_manager.py)
    ├─ update_track() → [track_created | track_updated]
    ├─ face_detected() → face_detected
    ├─ recognition_pending() → recognition_pending
    ├─ recognition_confirmed() → [person_entered | person_exited]
    ├─ unknown_confirmed() → [unknown_entered | unknown_exited]
    └─ cleanup_lost_tracks() → (no event, internal cleanup)
    ↓
EventPublisher.publish(event_type, payload)
    ├─ Filter: if event NOT in ALLOWED_EVENTS → return
    ├─ Pub/Sub: redis.publish(channel, json)
    └─ Stream: redis.xadd(stream_key, {event, payload, timestamp})
```

## 📊 Redis Storage

### Pub/Sub Channel
```
Channel: live-face-tracker:camera-events:{camera_code}
Message Format: {"event": "person_entered", "camera_code": "CAM_001", ...}
Behavior: Only active subscribers receive (no history)
TTL: Immediate (consumed or lost)
```

### Stream Key
```
Key: live-face-tracker:stream:camera-events:{camera_code}
Entry Format: {
    "event": "person_entered",
    "payload": "{...full payload...}",
    "timestamp": 1714900000000
}
TTL: Configurable (default: no auto-expiry)
Consumer Groups: Can be created for parallel processing
```

## 🚀 Usage Examples

### 1️⃣ Subscribe to Real-Time Events (Pub/Sub)
```python
# Node.js / Backend subscriber
import redis

r = redis.Redis(host='localhost', port=6379, decode_responses=True)
pubsub = r.pubsub()

pubsub.subscribe('live-face-tracker:camera-events:CAM_001')

for message in pubsub.listen():
    if message['type'] == 'message':
        event = json.loads(message['data'])
        print(f"Event: {event['event']} → {event['person_id']}")
```

### 2️⃣ Read Event History from Stream
```python
# Get last 10 events
events = redis_client.xrevrange(
    'live-face-tracker:stream:camera-events:CAM_001',
    count=10
)

for event_id, event_data in events:
    print(f"ID: {event_id}, Data: {event_data}")
```

### 3️⃣ Create Consumer Group (For multiple workers)
```python
# Create consumer group
redis_client.xgroup_create(
    'live-face-tracker:stream:camera-events:CAM_001',
    'face-events-group',
    id='0',  # Start from beginning
    mkstream=True
)

# Read as consumer
messages = redis_client.xreadgroup(
    groupname='face-events-group',
    consumername='worker-1',
    streams={'live-face-tracker:stream:camera-events:CAM_001': '>'},
    count=5
)
```

### 4️⃣ Stream Cleanup (Optional TTL)
```python
# Auto-trim to last 10,000 entries (MAXLEN)
redis_client.xtrim(
    'live-face-tracker:stream:camera-events:CAM_001',
    maxlen=10000,
    approximate=False
)
```

## ⚙️ Configuration Options

### To add other events to streams (future):

```python
# In publisher.py, modify:
STREAM_EVENTS = {
    "person_entered",
    "person_exited", 
    "unknown_entered",
    "unknown_exited",
    # "face_detected",          # Uncomment if needed (high volume)
    # "recognition_pending",    # Uncomment if needed
}
```

### To add events to Pub/Sub only (non-critical):

```python
# In publisher.py, modify:
ALLOWED_EVENTS = {
    "person_entered",
    "person_exited",
    "unknown_entered",
    "unknown_exited",
    # "face_detected",          # Pub/Sub only, no stream
}

# Remove from STREAM_EVENTS if also added
```

### To configure stream retention (Redis commands):

```bash
# Set max length to 100k entries
XTRIM live-face-tracker:stream:camera-events:CAM_001 MAXLEN 100000

# Set max age (requires Redis 6.2+)
XTRIM live-face-tracker:stream:camera-events:CAM_001 MINID 1714800000000
```

## 📌 Key Differences

| Aspect | Pub/Sub | Stream |
|--------|---------|--------|
| **Persistence** | ❌ None | ✅ Yes |
| **History** | ❌ No | ✅ Yes |
| **Replay** | ❌ No | ✅ Yes |
| **Consumer Groups** | ❌ No | ✅ Yes |
| **Real-time** | ✅ Yes | ✅ Yes |
| **Memory** | Low | Higher (stores history) |
| **Use Case** | Live dashboards | Audit log, replay, parallel processing |
